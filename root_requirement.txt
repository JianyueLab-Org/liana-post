Liana Pacific Postal System 微服务架构设计报告（V2）
一、项目背景
Liana Pacific Postal System（以下简称 Liana）是面向太平洋岛屿国家邮政场景设计的分布式邮政业务系统。
与传统快递系统不同，本系统需要解决以下特殊问题：
•	岛屿之间网络连接不稳定；
•	部分支局长期处于离线自治状态；
•	邮政业务存在“收寄—封发—转运—投递”的多级流转链路；
•	需要支持国际邮政体系的标准化邮件编码；
•	需要满足邮政行业对总包、交接、封发及轨迹追踪的业务要求。
为避免传统单体系统难以扩展的问题，本项目采用微服务架构设计，并结合领域驱动设计（DDD）思想进行业务边界划分。
注意：与邮件号有关的开发需准许upu S10标准，确保系统未来可兼容国际邮件交换要求，参考UpuBarcodeUtil类实现邮件号生成规则。
项目整体功能简介
本项目围绕邮政业务全链路提供端到端支撑，覆盖基础主数据、邮件收寄、封发分拣、运输交接、物流轨迹、离线同步以及统一前端工作台等核心能力。
1. 认证与权限管理：提供用户登录、JWT 令牌、用户/角色/权限管理，以及前端菜单与操作权限控制。
2. 邮政基础数据管理：维护邮政网点、转运中心、国际交换局、运输路线、国家信息和服务类型等基础主数据。
3. 邮件收寄与台账管理：支持邮件创建、列表、详情、状态流转、运单号生成，以及寄件人、收件人等信息维护。
4. 封发与分拣作业：支持总包/邮袋生成、批次管理、交接确认、接收勾核、开拆作业、路由计算、再次封发和出口处理。
5. 物流轨迹查询：支持按运单号与事件维度查询邮件轨迹，记录收寄、分拣、封发、到站、投递等关键事件。
6. 运输资源与调度：支持运输资产、运输线路、运输班次和运输任务管理，为跨节点转运提供支撑。
7. 离线同步与补偿：通过 Outbox、同步任务和重试记录处理网络不稳定场景下的数据暂存、同步与补偿。
8. 前端统一工作台：提供登录、工作台、收寄、邮件台账、封发、分拣、轨迹、基础数据、运输、系统管理、同步监控和投递等页面入口。

二、总体架构设计
系统采用：
•	Spring Cloud Alibaba（Nacos）
•	Feign
•	Spring Boot
•	MySQL
•	Redis
•	RabbitMQ（可选）
•	Docker
构建微服务体系。
系统主要由六个核心服务组成：
1.	liana-auth-service
2.	liana-facility-service
3.	liana-oms-service
4.	liana-dispatch-service
5.	liana-tracking-service
6.	liana-syncer
2.4 运行环境与依赖
基础运行环境
- JDK 17
- Maven 3.8+（用于多模块构建与打包）
- Node.js LTS（用于 `liana-web` 前端开发与构建）
- MySQL 8.0.x
- Redis 6.x/7.x
- Nacos 2.x
- 推荐在本机或同一内网环境提供统一的 `127.0.0.1:8848` 注册与配置中心

后端核心依赖
- Spring Boot 3.2.4
- Spring Cloud 2023.0.1
- Spring Cloud Alibaba 2023.0.1.0
- Spring Cloud Gateway
- Spring Cloud OpenFeign
- Spring Cloud LoadBalancer
- MyBatis-Plus 3.5.5
- MySQL Connector/J 8.0.33
- Spring Data Redis
- JJWT 0.11.5
- fastjson2 2.0.40
- Lombok

模块级依赖关系
- `liana-common`：JWT、通用模型、常量与公共工具，所有业务服务均依赖
- `liana-gateway`：Nacos Discovery/Config、Gateway、OpenFeign、LoadBalancer、Actuator
- `liana-auth-service`：Web、Validation、JDBC、MyBatis-Plus、Nacos、OpenFeign、LoadBalancer、Security Crypto、MySQL
- `liana-facility-service`：Web、Validation、MyBatis-Plus、Nacos、OpenFeign、LoadBalancer、MySQL
- `liana-oms-service`：Web、Validation、JDBC、Redis、MyBatis-Plus、Nacos、OpenFeign、LoadBalancer、MySQL
- `liana-dispatch-service`：Web、Validation、JDBC、Redis、MyBatis-Plus、fastjson2、Nacos、OpenFeign、LoadBalancer、MySQL
- `liana-sorting-service`：Web、Validation、JDBC、Redis、MyBatis-Plus、Nacos、OpenFeign、LoadBalancer、MySQL，并依赖 `liana-oms-service`
- `liana-tracking-service`：Web、Validation、Redis、MyBatis-Plus、Nacos、OpenFeign、LoadBalancer、MySQL
- `liana-syncer`：Spring Boot Starter、JDBC、Validation、MyBatis-Plus、Nacos、OpenFeign、LoadBalancer、MySQL
- `liana-transport-service`：Web、Validation、MyBatis-Plus、Nacos、OpenFeign、LoadBalancer、MySQL

前端依赖
- Vue 3.5.13
- Vue Router 4.5.0
- Pinia 3.0.4
- Axios 1.17.0
- Vite 6.3.5
- `@vitejs/plugin-vue` 5.2.4
- Tailwind CSS 4.1.8
- `@tailwindcss/vite` 4.1.8
- 前端开发服务器默认代理到 `http://127.0.0.1:8080`

运行配置约束
- 各服务默认通过 Nacos 读取配置，`NACOS_SERVER_ADDR` 默认值为 `127.0.0.1:8848`
- 数据库按服务拆分，默认库名包括 `auth_db`、`facility_db`、`oms_db`、`dispatch_db`、`sorting_db`、`tracking_db`、`transport_db`、`syncer_db`
- Redis 默认使用 `127.0.0.1:6379`，`oms`、`dispatch`、`sorting`、`tracking` 会读取该实例
- `tracking` 的 SQL 初始化脚本位于 `db/tracking_db.sql`
- `tracking` 与 `syncer` 当前默认都使用 `8085` 端口，联调时需要手工调整其一的端口配置
- 架构中预留了 RabbitMQ，但当前仓库未将其作为必需运行依赖

服务定位
统一身份认证与权限管理服务。
核心职责
负责：
•	用户登录
•	JWT Token签发
•	用户权限校验
•	角色管理
•	单点登录支持
主要实体

Role
Permission
典型角色

邮局经理（MANAGER）

五、数据库原则
采用 Database Per Service 原则。
每个服务拥有独立数据库。
AuthDB
FacilityDB
OmsDB
DispatchDB
TrackingDB
禁止跨服务直接访问数据库。
统一通过：
REST API
或
消息事件
进行通信。

六、架构优势
1.	业务边界清晰
邮件、封发、轨迹、设施各自独立。
2.	符合DDD思想
按照领域模型拆分，而非按照组织机构拆分。
3.	支持岛屿离线自治
断网情况下支局仍可正常营业。
4.	具备扩展能力
未来可扩展：
国际交换局
海运管理
无人机投递
UPU国际邮件交换
等高级业务场景。
5.	适合作为教学与毕业设计项目
既体现微服务架构思想，又具有真实邮政行业业务背景。

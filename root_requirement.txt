Liana Pacific Postal System 微服务架构设计报告（V2）
一、项目背景
Liana Pacific Postal System（以下简称 Liana）是面向太平洋岛屿国家邮政场景设计的分布式邮政业务系统。
与传统快递系统不同，本系统需要解决以下特殊问题：
•	岛屿之间网络连接不稳定；
•	部分支局长期处于离线自治状态；
•	邮政业务存在“收寄—封发—转运—投递”的多级流转链路；
•	需要支持国际邮政体系的标准化邮件编码；
•	需要满足邮政行业对总包、交接、封发及轨迹追踪的业务要求。
为避免传统单体系统难以扩展的问题，本项目采用微服务架构设计，并结合领域驱动设计（DDD）思想进行业务边界划分。
注意：与邮件号有关的开发需准许upu S10标准，确保系统未来可兼容国际邮件交换要求，参考UpuBarcodeUtil类实现邮件号生成规则。

二、总体架构设计
系统采用：
•	Spring Cloud Alibaba（Nacos）
•	Feign
•	Spring Boot
•	MySQL
•	Redis
•	RabbitMQ（可选）
•	Docker
构建微服务体系。
系统主要由六个核心服务组成：
1.	liana-auth-service
2.	liana-facility-service
3.	liana-oms-service
4.	liana-dispatch-service
5.	liana-tracking-service
6.	liana-syncer

三、微服务划分
3.1 liana-auth-service
服务定位
统一身份认证与权限管理服务。
核心职责
负责：
•	用户登录
•	JWT Token签发
•	用户权限校验
•	角色管理
•	单点登录支持
主要实体
User
Role
Permission
典型角色
邮局营业员（CLERK）
邮局经理（MANAGER）
转运中心操作员（SORTER）
系统管理员（ADMIN）
服务特点
整个系统唯一认证入口。
其他服务不保存用户密码。

3.6 liana-syncer
服务定位
离线自治同步组件。
核心职责
负责跨海岛环境的数据同步。
不属于业务服务
Syncer不参与任何邮政业务处理。
其职责仅包括：
消息暂存
断网缓存
失败重试
数据同步
网络恢复补偿
核心实体

↓

↓

↓

与

五、数据库原则
采用 Database Per Service 原则。
每个服务拥有独立数据库。
AuthDB
FacilityDB
OmsDB
DispatchDB
TrackingDB
禁止跨服务直接访问数据库。
统一通过：
REST API
或
消息事件
进行通信。

六、架构优势
1.	业务边界清晰
邮件、封发、轨迹、设施各自独立。
2.	符合DDD思想
按照领域模型拆分，而非按照组织机构拆分。
3.	支持岛屿离线自治
断网情况下支局仍可正常营业。
4.	具备扩展能力
未来可扩展：
国际交换局
海运管理
无人机投递
UPU国际邮件交换
等高级业务场景。
5.	适合作为教学与毕业设计项目
既体现微服务架构思想，又具有真实邮政行业业务背景。
